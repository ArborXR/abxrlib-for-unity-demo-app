# PICO dual-SDK setup (demo app)

The demo project uses **both** `com.unity.xr.picoxr` (PICO Unity Integration) and `com.unity.xr.openxr.picoxr` (OpenXR PICO) as local `file:` packages. Unity cannot compile two copies of the same assembly; PICO also has CS0433 issues when both are present. **Do not** hand-edit the standard SDK zips—run the setup script instead.

## Repo layout (recommended)

Place this repo next to the **abxrlib** package and your **extracted PICO SDK folders** under one workspace (e.g. `arborxr/`):

```text
arborxr/
  abxrlib-for-unity/                          # AbxrLib (local package)
  abxrlib-for-unity-demo-app/                 # this repo
  PICO Unity Integration SDK-3.2.0-20250707/  # unzip from PICO (source; unmodified)
  Unity OpenXR IntegrationSDK-1.4.0-20250407/ # unzip from PICO (source; unmodified)
  PICO Unity Integration SDK-demo-app/        # generated by script (gitignored optional)
  Unity OpenXR IntegrationSDK-demo-app/       # generated by script
```

`Packages/manifest.json` uses **relative** `file:../../…` paths. Unity resolves local packages from the **`Packages/`** folder, so one `..` is the project root and **`../..`** is the workspace parent (e.g. `arborxr/`) where the generated SDK folders live.

## One-time (or after upgrading SDK zips)

From **this repo root**:

```bash
bash tools/pico-setup-demo-sdks.sh
```

This will:

1. **Copy** each standard SDK folder to the `*-demo-app` names (replacing any previous copy).
2. **Remove** duplicate `Enterprise/` + `Platform/` from the **OpenXR** copy only.
3. **Patch** the **Integration** copy for CS0433 (C# + asmdef).
4. **Patch** `Runtime/Windows/x86_64/applogrs.dll(.lib)` **PluginImporter** so they are **not** enabled for **Android** (PICO’s default `Any` causes “not supported on Android” at build time).
5. **macOS only:** clear quarantine on `SpatialAudio/Plugins/OSX` so the Editor loads dylibs without Gatekeeper prompts.

### Environment overrides

If your folders live elsewhere or use different version folder names:

```bash
export WORKSPACE_ROOT=/path/to/workspace
export PICO_INTEGRATION_SOURCE="/path/to/PICO Unity Integration SDK-3.2.0-20250707"
export OPENXR_SOURCE="/path/to/Unity OpenXR IntegrationSDK-1.4.0-20250407"
export PICO_INTEGRATION_DEMO="/path/to/PICO Unity Integration SDK-demo-app"
export OPENXR_DEMO="/path/to/Unity OpenXR IntegrationSDK-demo-app"
bash tools/pico-setup-demo-sdks.sh
```

## Windows

Use **Git Bash** (from Git for Windows) so `bash`, `cp`, `rm`, and `python3` behave like on macOS/Linux. Ensure Python 3 is on `PATH`.

## AbxrLib path

If you switch AbxrLib to a local package, use `file:../../abxrlib-for-unity` (same `Packages/`-relative rule). The default manifest may use the git URL instead.

## Why this project behaves differently from other PICO builds

Most PICO sample apps use **one** SDK path: either the **PICO Unity Integration** (`com.unity.xr.picoxr`) **or** **OpenXR + PICO features**, often from a single package install.

This demo is **not** that layout. It intentionally depends on **two** local packages that must be **generated** from official PICO zips by `pico-setup-demo-sdks.sh`, because AbxrLib needs types from the Integration package (e.g. `PXR_Enterprise`, TOB) while the player uses **OpenXR** as the Android XR loader. If the script was skipped, paths are wrong, or an old copy of `*-demo-app` folders is on disk, you can get subtle runtime issues (enterprise bind, QR, etc.) even though another “plain PICO” project on the same headset works.

## Verification checklist (before blaming AbxrLib)

1. **Workspace layout** — From `abxrlib-for-unity-demo-app/`, the parent folder must contain:
   - `abxrlib-for-unity/` (if using the local `file:` path in `Packages/manifest.json`)
   - `PICO Unity Integration SDK-demo-app/` and `Unity OpenXR IntegrationSDK-demo-app/` after running the setup script  
   If you only cloned the demo repo into an empty folder, `file:../../…` packages **do not resolve** until you add siblings and run the script.

2. **Run** `bash tools/pico-setup-demo-sdks.sh` after every upgrade of the **source** PICO zip folders, then reopen Unity so it reimports packages.

3. **Android Player — Scripting Define Symbols** — In **Project Settings → Player → Android**, ensure `PICO_ENTERPRISE_SDK_3` is present (the demo usually has it; without it, `QRCodeReaderPico` is compiled out and PICO enterprise QR is unavailable).

4. **XR Plug-in Management** — Android should use the **OpenXR** loader (as in this repo’s `Assets/XR/XRGeneralSettings.asset`). That is expected here; do not switch to the legacy PXR loader unless you are following PICO’s single-SDK docs and accept diverging from this demo.

5. **QR scan returns to the app** — The OS may foreground `com.pvr.tobservice` (QR Scanner) and then return to your app. If the scan string never reaches AbxrLib, check logcat for `[AbxrLib] PICO QR:` lines after a scan (payload length / main-thread processing). Missing lines usually mean the SDK callback did not run or packages/native code are mismatched relative to the device OS.

6. **Android 11+ package visibility** — This repo ships **`Assets/Plugins/Android/AndroidManifest.xml`** with a `<queries>` entry for `com.pvr.tobservice` so the merged APK can resolve the PICO TOB QR package when the platform returns control to Unity. Rebuild the Android player after pulling; do not delete that file for PICO QR testing.
